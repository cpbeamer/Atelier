# Atelier v0.5 — "The Full Workflow" Design Spec

**Date:** 2026-04-19
**Author:** Cary Beam
**Status:** Approved for implementation
**Phase:** v0.5

---

## Overview

v0.5 builds on the MVP (v0.1) by adding multi-step workflows, a draggable terminal grid, human milestone gates, worktree sandboxing, and first-class Python workflow support. The result is a desktop app where users can run adversarial multi-agent workflows with visible PTY terminals, pause for human approval at key points, and have workflow state survive crashes.

## 1. Multi-Step Workflow DSL

### TypeScript (`defineWorkflow`)

Workflows live at `.atelier/workflows/*.workflow.ts`.

```ts
import { defineWorkflow, callAgent, milestone, condition } from '@atelier/workflow';
import { z } from 'zod';

export const featureWorkflow = defineWorkflow({
  name: 'feature',
  input: z.object({ signal: z.string() }),

  run: async (ctx, input) => {
    // Step 1: PM Specialist drafts a proposal
    const proposal = await callAgent(ctx, 'PM Specialist', { signal: input.signal });

    // Step 2: PM Validator tears it apart
    const verdict = await callAgent(ctx, 'PM Validator', { proposal });

    // Step 3: Human gate — workflow blocks here until user approves/rejects
    const decision = await milestone(ctx, 'PM Proposal Review', {
      proposal,
      verdict,
    });

    if (decision.verdict !== 'Approved') {
      return { status: 'rejected', verdict: decision };
    }

    // Step 4: Architect designs
    const design = await callAgent(ctx, 'Architect', { proposal });
    // ... continue
  },
});
```

### Python (`@workflow.defn`)

Workflows live at `.atelier/workflows/*.workflow.py`.

```python
from atelier_workflow import workflow, call_agent, milestone
from pydantic import BaseModel

class FeatureInput(BaseModel):
    signal: str

@workflow.defn
class FeatureWorkflow:
    @workflow.run
    async def run(self, input: FeatureInput):
        proposal = await call_agent("PM Specialist", {"signal": input.signal})
        verdict = await call_agent("PM Validator", {"proposal": proposal})

        decision = await milestone("PM Proposal Review", {
            "proposal": proposal,
            "verdict": verdict,
        })

        if decision["verdict"] != "Approved":
            return {"status": "rejected", "verdict": decision}

        design = await call_agent("Architect", {"proposal": proposal})
        # ...
```

### Primitives

| Primitive | TS | Python | Behavior |
|---|---|---|---|
| `callAgent(name, input)` | ✓ | ✓ | Invokes named agent, returns result |
| `milestone(name, payload)` | ✓ | ✓ | Blocks, sends signal to UI, awaits human decision |
| `condition(predicate)` | ✓ | ✓ | Blocks on a boolean Temporal condition |

### Hot Reload

- Bun worker uses `chokidar` to watch `.atelier/workflows/*.workflow.ts`
- Python worker uses `watchfiles` to watch `.atelier/workflows/*.workflow.py`
- On change: register/re-reregister workflow with Temporal
- Active runs continue using the code version they started with (Temporal preserves state)

### Routing

The **WorkflowDispatcher** in `backend/` is the single seam that routes by language:

```ts
// pseudo-code
const taskQueue = wf.language === 'typescript'
  ? `atelier-${projectId}-ts`
  : `atelier-${projectId}-py`;
```

UI is language-agnostic — reads workflow metadata, dispatches to the right queue, renders terminals uniformly.

## 2. Terminal Grid

### Layout

- **Grid engine:** `react-grid-layout` (drag-and-drop tileable grid)
- **Default:** 1 column, 1 pane. Users drag to create new rows/columns.
- **Pane limit:** 8 concurrent panes per project (configurable)
- **Resize handles:** on all edges and corners

### Per-Pane: xterm.js

- Renders PTY output streamed from main process via IPC events
- Header bar: agent name, status badge (running/exited/killed), close button
- ANSI color support, clickable links via `xterm-addon-web-links`
- Fit addon matches pane size

### PTY Lifecycle

1. Backend `PTYManager.spawn()` creates a new node-pty process
2. PTY output streamed to renderer via `terminal-output` IPC event
3. Renderer writes to xterm.js instance
4. User keyboard input → `terminal-input` IPC → pty master write
5. On workflow step complete: pane stays open (user can scroll) or auto-closes based on agent config

### Dynamic Tiling UX

- **Add pane:** "+" button in grid toolbar → select agent → pane appears
- **Remove pane:** click "×" on pane header
- **Reorder:** drag pane by header
- **Resize:** drag edges/corners
- **Detach:** double-click header → pop out to new window

## 3. Human Milestones

### Signal Flow

```
Workflow calls milestone(name, payload)
    ↓
Milestone activity writes to SQLite: milestones table (status='pending')
    ↓
Main process sends 'milestone-created' IPC event to renderer
    ↓
Renderer shows notification + inbox badge
    ↓
User reviews in MilestoneInbox → clicks Approve/Reject/Defer
    ↓
Renderer calls main process: milestoneDecision(milestoneId, decision)
    ↓
Main process sends Temporal signal: 'milestone-decision'
    ↓
Workflow's condition() resolves
    ↓
Workflow resumes
```

### Milestone Table Schema

```sql
milestones (
  id TEXT PK,
  run_id TEXT FK,
  type TEXT,             -- 'milestone'
  status TEXT,           -- 'pending' | 'approved' | 'rejected' | 'timed-out'
  payload_json TEXT,
  created_at INTEGER,
  decided_at INTEGER,
  decided_by TEXT,       -- 'user' | 'auto-timeout'
  decision_reason TEXT
)
```

### Inbox UI

- Sidebar shows inbox icon with pending count badge
- Click opens MilestoneInbox panel (slides in from right)
- Shows: workflow name, milestone name, created-at, payload (formatted JSON)
- Actions: **Approve** (green) / **Reject** (red, requires reason) / **Defer** (grey, moves to end of queue)

### Auto-Timeout

- Configurable per-milestone: `timeout: '7d'` (default 7 days)
- On timeout: auto-reject with `reason: 'milestone-timeout'`, notify user

## 4. Worktree Sandbox

### Layout

```
~/.atelier/worktrees/{project}/{run-id}/
```

### Lifecycle

1. On workflow start: `git worktree add` a new branch `atelier/{workflow}/{run-id}`
2. Agents operate on worktree files only
3. On milestone approval: optional promotion (merge to main or push branch + open PR)
4. On workflow complete/fail: worktree optionally retained for review

### Git Config

- Worktree has no remote configured (cannot push without explicit enable)
- Agents run with `--dangerously-skip-permissions` but confined to worktree

## 6. Sidecar Lifecycle (Temporal Binary)

### Startup

On app launch, main process:

1. Ensure Temporal binary is present at `~/.atelier/temporal/temporal`
2. Create data dir `~/.atelier/temporal/data/`
3. Spawn `temporal server start-dev --db-filename atelier.db --port 7466 --http-port 7467 --ui-port 8466`
4. Wait for health check (poll `localhost:7466/health` or log line "Temporal server is up")
5. Connect workers

### Shutdown

On app quit: SIGTERM Temporal child process, clean exit.

### Per-Project Namespace

On first run for a project: `temporal namespace register atelier-{slug}`

## 7. Python Worker Architecture

### Runtime Setup

1. First time a project opens with `.workflow.py` files:
2. Atelier creates `<project>/.atelier/.venv-py/` via `uv venv`
3. Installs: `temporalio>=1.8`, `pydantic>=2.0`, `atelier-workflow`
4. Plus any deps from `.atelier/python-requirements.txt`

### Worker Process

```ts
// Backend spawns:
spawn('uv', ['run', '--project', '.atelier/.venv-py', 'python', '-m', 'atelier_worker'], {
  cwd: projectRoot,
  env: { TEMPORAL_ADDRESS: '127.0.0.1:7466', TEMPORAL_NAMESPACE: `atelier-${slug}` }
});
```

### atelier_workflow Package

PyPI package providing:
- `workflow` — decorator equivalent of `defineWorkflow`
- `call_agent(name, input)` — invokes named agent (terminal or direct LLM)
- `milestone(name, payload)` — blocks until human decision
- `child_workflow(name, args)` — cross-language via Temporal child workflow

## 8. Error Handling

| Scenario | Behavior |
|---|---|
| Agent PTY exits non-zero | Show red status in pane header; continue workflow (some agents expect to fail) |
| Temporal sidecar crashes | Main process restarts it; workers reconnect; active workflows resume |
| Python venv not set up | Block with setup wizard; offer to auto-create |
| Milestone timeout | Auto-reject, notify user, workflow continues with rejection reason |
| Max concurrent agents reached | Queue new agents; show "waiting" badge in pane |

## 12. Build Order

1. **Terminal grid** — xterm.js panes with draggable layout, PTY streaming
2. **Multi-project sidebar** — project switching, namespace isolation
3. **Workflow DSL + Bun worker** — `defineWorkflow`, `callAgent`, hot-reload
4. **Python worker** — uv venv setup, `@workflow.defn` support
5. **Milestone signals** — milestone activity, inbox UI, Approve/Reject/Defer
6. **Worktree sandbox** — git worktree lifecycle per run
